# Internationalization (i18n)

The site is translated into **11 languages** — English (source), Spanish, Chinese,
Korean, French, German, Dutch, Italian, Portuguese, Russian, Japanese — and each
language is served at **its own URL** so search engines index every market. Prices are
**always shown in USD** regardless of language.

- **English** is the default, served at the site root (`/`, `/guides.html`, …).
- **Every other language** is served under a URL prefix: `/es`, `/fr/guides.html`,
  `/de/contact.html`, … **rendered server-side** from the same language JSON, so a
  crawler (or a no-JS client) sees fully-translated HTML with a distinct, canonical,
  crawlable URL per language.

> **Why the URL matters.** Previously the translation was 100% client-side on a single
> URL, so Google only ever indexed English — the translated content was invisible to
> search in the other 10 markets. Serving each language at its own URL (with `hreflang`)
> is what makes those markets findable. This is a **serving/routing** layer on top of
> the existing translation JSON — not a second copy of the content.

## The pieces

### Config — the single source of truth

[`lib/i18n/locales.js`](../../lib/i18n/locales.js) defines **which languages get a URL
prefix** (`LOCALES`) and **which pages are localized** (`LOCALIZED_PAGES`), plus the
helpers (`localizedUrl`, `buildHreflangCluster`, …) everything else derives from. Change
the language or page set **here** and the router, sitemap, and hreflang all follow.

The localized page set is the indexable marketing/legal pages: `/`, `ai-designer.html`,
`masking-studio.html`, `stagify-plus.html`, `enterprise.html`, `guides.html`,
`contact.html`, `/status`, `privacy.html`, `terms.html`. Deliberately **excluded**: the
blog (articles aren't in the translation JSON — English-only), `faq.html` (canonicalizes
to `/index.html#faq`), and the auth/app pages (noindex).

### Server — request-time rendering

[`routes/i18n.js`](../../routes/i18n.js) registers `/<prefix>` and `/<prefix>/<page>` for
every locale × page and renders them through
[`lib/i18n/render-page.js`](../../lib/i18n/render-page.js), which:

- applies the same `data-lang*` attributes + `<title>` / JSON-LD the client uses, so the
  body ships already-translated;
- sets `<html lang>` + a `data-locale` marker the client reads;
- injects **`<base href="/">`** so every relative asset URL (including ones scripts build
  at runtime — logo images, the heic2any worker, the `languages/<lang>.json` fetch)
  resolves against the site root, not `/<prefix>/`;
- writes a self-referential `<link rel="canonical">`, the full `hreflang` cluster, and
  localized `og:*` / `twitter:*` tags;
- rewrites internal `<a href>` nav (and bare `#…` anchors) to stay inside the prefix,
  because `<base href="/">` would otherwise send them to the English root.

Rendering is a pure string transform (no DOM library) and is memoized per
`(prefix, page)`; `DEBUG_MODE` bypasses the cache for local dev. Caches are
process-lifetime, so a content change is picked up on the next deploy/restart.

### Client — the URL is authoritative

Three ES-module scripts in `public/scripts/`, on every page:

| Script | Role |
|---|---|
| `language-detect.js` | Picks the language: **on a localized URL the URL wins**; otherwise `localStorage.selectedLanguage`, else `navigator.languages`. Persists the choice and sets `<html lang>`. |
| `language-loader.js` | Fetches `languages/<lang>.json`, applies it to the DOM, keeps `<title>`/JSON-LD in sync, re-applies to dynamically-added nodes (`MutationObserver`), and re-localizes links after each apply. Exposes `window.LanguageSystem`. |
| `language-switcher.js` | The custom flag-dropdown UI over the hidden native `#language-select`. |
| `i18n-routing.js` | Shared helper: maps prefix↔language, and computes the localized URL for the switcher and in-app redirects. Mirrors `lib/i18n/locales.js`. |

Key behaviors that differ from a pure client-side system:

- **Switching language navigates** to the localized URL (a full load the server renders
  in-language) instead of swapping strings in place — so the URL, canonical, and
  hreflang always match what the visitor sees.
- **`localizeLinks()`** re-prefixes internal links after every translation apply, because
  applying `[data-lang-html]` resets links inside translated rich text to their raw JSON
  form (bare `#…` anchors, `terms.html`, …) which would break under `<base href="/">`.
- The **English root still client-swaps** to the visitor's stored language (invisible to
  crawlers, which have no `localStorage`), preserving the previous auto-language UX.

If a language file fails to load, the loader falls back to a tiny built-in English object
so the page never renders empty.

## Marking text for translation

Add a `data-*` attribute whose value is a **dot-path key** into the language JSON. The
server (SSR) and client apply these identically:

| Attribute | Effect | Example |
|---|---|---|
| `data-lang="key"` | Sets `textContent` (or `placeholder` on text `<input>` / `<textarea>` — client-side only). | `<h1 data-lang="hero.title.free">Free</h1>` |
| `data-lang-html="key"` | Sets `innerHTML` — use when the string contains markup. | `<p data-lang-html="home.masking.subtitle">…</p>` |
| `data-lang-attr="key\|attr"` | Sets an arbitrary attribute. | `<img data-lang-attr="navigation.logoAlt\|alt">` |

Also handled automatically: `<title data-lang="meta.title">`, and the page's
`application/ld+json` `name`/`description`/`keywords` (refreshed from the page's title /
description / `meta.keywords` keys). Always keep the **English** text inline as the
default (it's the root-URL content and the pre-JS fallback).

> **Form fields are client-only.** `<input>`/`<textarea>` with `data-lang` set a runtime
> *placeholder*, not content — the SSR renderer skips them so it never pre-fills the
> field with placeholder text; the client localizes them on load.

## The language files

- Location: `public/languages/<language>.json` (e.g. `english.json`, `french.json`).
- **`english.json` is the source of truth** for the key set. Every other file mirrors its
  structure; keys are the dot-paths referenced by `data-lang*` attributes.
- A test enforces that **every language file is valid, non-empty JSON and covers the keys
  in `english.json`** (see [`testing.md`](testing.md)) — a missing key fails CI.

## The build step — run after changing locales/pages

`sitemap.xml` (per-language `<url>` entries + `xhtml:link` alternates) and the **hreflang
cluster baked into each static English page** are generated from `lib/i18n/locales.js` by:

```bash
node scripts/build-i18n-seo.js
```

**Rerun it after** changing the language set, the page set, or a page's canonical. It's
idempotent and preserves each file's line endings. A drift-guard test
([`test/i18n.test.js`](../../test/i18n.test.js)) fails if the committed sitemap or English
hreflang is stale — so a forgotten rebuild blocks the deploy.

## Adding or changing a string

1. Add the key + English value to `english.json`.
2. Add the **same key** to every other `languages/*.json` (the coverage test fails on a
   gap).
3. Reference it from the HTML with `data-lang` / `data-lang-html` / `data-lang-attr`,
   keeping the English text inline as the default.

No rebuild is needed for a string change — only the JSON and HTML.

## Adding a new language

The language is referenced by its lowercase name (e.g. `portuguese`) and a short URL
prefix (e.g. `pt`). Keep these in sync:

1. **`public/languages/<language>.json`** — a full translation of `english.json`.
2. **`lib/i18n/locales.js`** — add a `LOCALES` entry (`prefix`, `lang`, `hreflang`,
   `bcp47`, `ogLocale`, `label`). This drives the routes, sitemap, and hreflang.
3. **`public/scripts/i18n-routing.js`** — add the prefix↔language entries
   (`PREFIX_TO_LANG` / `LANG_TO_PREFIX`).
4. **`language-detect.js`** — add the browser-tag → language mapping in `toSupported()`
   and the `BCP47` entry.
5. **`language-switcher.js`** — add the `FLAGS` and `BCP47` entries, and the `<option>`
   in each page's `#language-select` + the custom switcher list.
6. **The flag asset** under `public/media-webp/flags/`.
7. **Run `node scripts/build-i18n-seo.js`** to add the language to the sitemap and every
   page's hreflang cluster.

> The switcher markup is repeated per page — grep for an existing language name (e.g.
> `italian`) to find every spot.

## Adding a page to the localized set

Add it to `LOCALIZED_PAGES` in [`lib/i18n/locales.js`](../../lib/i18n/locales.js)
(`path`, `file`, `lastmod`, `changefreq`, `priority`) and run
`node scripts/build-i18n-seo.js`. The route, sitemap entries, and hreflang follow
automatically.

## BCP-47 mapping

`<html lang>` uses BCP-47 codes, defined in `lib/i18n/locales.js` and mirrored in
`language-detect.js` / `language-switcher.js`:

```
english→en  spanish→es  chinese→zh-Hans  korean→ko  french→fr  german→de
dutch→nl  italian→it  portuguese→pt-BR  russian→ru  japanese→ja
```
