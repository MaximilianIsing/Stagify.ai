# Internationalization (i18n)

The UI is translated **entirely client-side**. There is no server involvement and no
build step: each page ships English text in the HTML, and a small script swaps it for
the visitor's language at load time from a JSON file.

Currently **11 languages**: English (source), Spanish, Chinese, Korean, French, German,
Dutch, Italian, Portuguese, Russian, Japanese. Prices are **always shown in USD**
regardless of language.

## How it works

Three scripts in `public/scripts/`, loaded as ES modules (`type="module"`, so deferred
by default) on every page:

| Script | Role |
|---|---|
| `language-detect.js` | On first visit, picks a supported language from the browser's `navigator.languages`, persists it to `localStorage.selectedLanguage`, and sets `<html lang>` before first paint. An explicit earlier choice always wins. |
| `language-loader.js` | The core runtime. Fetches `languages/<lang>.json`, applies it to the DOM, updates `<title>` + JSON-LD, and re-applies to dynamically added nodes via a `MutationObserver`. Exposes `window.LanguageSystem`. |
| `language-switcher.js` | The custom flag-dropdown UI. The hidden native `#language-select` stays the source of truth; changing it triggers a reload of strings. |

### Load sequence

1. `language-detect.js` resolves the language and writes `localStorage.selectedLanguage`.
2. `language-loader.js` reads that key (default `english`), `fetch`es
   `languages/<lang>.json`, and walks the DOM applying translations.
3. It adds `body.language-loaded` and fires a `languagechange` event; the switcher
   listens for it to sync its flag/label.
4. Switching language persists the new choice and re-applies strings — no page reload.

If a language file fails to load, the loader falls back to a tiny built-in English
object so the page never renders empty.

## Marking text for translation

Add a `data-*` attribute whose value is a **dot-path key** into the language JSON:

| Attribute | Effect | Example |
|---|---|---|
| `data-lang="key"` | Sets `textContent` (or `placeholder` on text `<input>` / `<textarea>`). | `<h1 data-lang="hero.title.free">Free</h1>` |
| `data-lang-html="key"` | Sets `innerHTML` — use when the string contains markup (e.g. an `<em>` or a link). | `<p data-lang-html="home.masking.subtitle">…</p>` |
| `data-lang-attr="key\|attr"` | Sets an arbitrary attribute. | `<img data-lang-attr="navigation.logoAlt\|alt">` |

Special cases the loader also handles automatically:

- `<title data-lang="meta.title">` updates the document title.
- The page's `application/ld+json` structured data has its `name` / `description` /
  `keywords` refreshed from `meta.title` / `meta.description` / `meta.keywords`.

The HTML should always contain the **English** text as a sensible default (it's what
shows before the loader runs and what search engines/no-JS clients see).

## The language files

- Location: `public/languages/<language>.json` (e.g. `english.json`, `french.json`).
- **`english.json` is the source of truth** for the key set. Every other file should
  mirror its structure; keys are the dot-paths referenced by `data-lang*` attributes.
- Values may contain HTML when targeted by `data-lang-html`.
- A test enforces that **every language file is valid, non-empty JSON and covers the
  keys in `english.json`** (see [`testing.md`](testing.md)) — a missing key fails CI.

## Adding or changing a string

1. Add the key + English value to `english.json`.
2. Add the **same key** to every other `languages/*.json` with its translation (don't
   leave it missing — the coverage test fails).
3. Reference it from the HTML with the appropriate `data-lang` / `data-lang-html` /
   `data-lang-attr` attribute, keeping the English text inline as the default.

## Adding a new language

Keep these places in sync (all reference the language by its lowercase name, e.g.
`portuguese`):

1. **`public/languages/<language>.json`** — a full translation of `english.json`.
2. **`language-detect.js`** — add the browser-tag → language mapping in `toSupported()`
   and the `BCP47` entry.
3. **`language-switcher.js`** — add the `FLAGS` (flag asset) and `BCP47` entries, and
   the option in the switcher markup on the pages.
4. **The flag asset** under `public/media-webp/flags/`.
5. The `<option>` in each page's `#language-select` and the custom switcher list.

> The add-a-language change touches several pages (the switcher markup is repeated per
> page). Grep for an existing language name (e.g. `italian`) to find every spot.

## BCP-47 mapping

`<html lang>` uses BCP-47 codes (for correct screen-reader pronunciation), mapped from
the internal language name in both `language-detect.js` and `language-switcher.js`:

```
english→en  spanish→es  chinese→zh-Hans  korean→ko  french→fr  german→de
dutch→nl  italian→it  portuguese→pt-BR  russian→ru  japanese→ja
```

Keep those two maps identical.
